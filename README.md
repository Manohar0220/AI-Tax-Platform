# AI Tax Platform

## 1. Platform Overview

AI Tax Platform is a role-aware tax preparation and collaboration prototype for individual taxpayers, business owners, tax preparers, reviewers, and firm administrators. Clients complete a short onboarding questionnaire, provide only the documents relevant to their situation, respond to preparer or reviewer requests, monitor progress, review their prepared return, and approve it. Tax preparers use AI-assisted review, source-document evidence, client communication, and workflow tools to prepare returns, and reviewers validate that work before the return moves to client approval and a simulated filing step.

This is  not a production tax-filing service. All data is fictional, and backend behaviour (authentication, storage, AI, tax calculations, and filing) is simulated in the browser.

## 2. Prototype Goals

The prototype demonstrates:

- A clear first-time onboarding experience
- Role-aware navigation and permissions
- Conditional document requirements based on onboarding answers
- Source-document traceability
- Contextual client and tax-team collaboration
- Shared return status and progress across all roles
- An actionable preparer dashboard
- Clear editable, verified, AI-generated, approval-required, and locked field states
- Search, filtering, and navigation across larger fake datasets
- Transparent and correctable AI recommendations

## 3. Demo Roles

Five roles are available from the demo login screen.

### Manohar — Individual Taxpayer
Completes personal-tax onboarding, sees only the documents relevant to the answers given, responds to preparer or reviewer requests, reviews return progress, and approves the finished return.

### Alex Rivera — Business Owner
Completes business onboarding, provides the relevant business records, responds to questions about income, employees, contractors, equipment, and expenses, and approves the business return.

### Maya Patel — Tax Preparer
Reviews newly submitted returns, verifies AI-extracted information, checks source evidence, corrects values, creates questions or document requests, communicates with clients, and submits returns for reviewer approval. Maya also has a personal return, so she can switch between the firm workspace and her own client workspace.

### Daniel Kim — Reviewer
Reviews corrections, warnings, evidence, and preparer decisions from the review queue. He can request changes or additional information, or approve the return.

### Priya Shah — Firm Administrator
Monitors returns, responsibilities, deadlines, workload, blockers, and status changes across the firm.


## 4. Recommended Demo Flow

One shared flow works for both Manohar (individual) and Alex (business).

### Client onboarding
1. Open the demo login page.
2. Sign in as **Manohar** for the Individual Taxpayer flow, or **Alex Rivera** for the Business Owner flow.
3. Complete the short onboarding questionnaire.
4. Onboarding answers control which documents, questions, and tax topics are relevant for the rest of the demo.
5. After onboarding, the return moves to **“Ready for preparer review.”**
6. Open **My Return** (or **Business Return**) to view the current stage, completed steps, next action, responsible role, any blocking items, and the progress timeline.
7. At this stage the client has no preset conversations or requests — nothing appears unless it is created later in the demo.

### Tax preparer review
1. Switch the demo user to **Maya Patel** (Tax Preparer).
2. Open the **Returns** page (or the dashboard queue).
3. The newly onboarded Manohar or Alex return appears near the top with a **New / ready-for-review** indicator.
4. Open the return.
5. On the **Overview** tab, review the current stage, next action, responsible role, completed steps, preparer, reviewer, blockers, and recent activity.
6. Open the **Review** tab.
7. AI recommendations are shown **inside Review**, not in a separate AI tab.
8. The AI recommendations are **mocked** — they are seeded data, not generated by a real LLM.
9. Review a recommendation to see what the AI found, why it matters, its confidence/uncertainty, supporting evidence, and the recommended action.
10. Use **Open supporting evidence** to jump to the source document / traceability view.
11. Demonstrate one or more actions: **Accept**, **Correct value**, **Dismiss**, **Ask client**, **Request document**, or **Escalate**.
12. Requests and conversations are not shown beforehand — they appear only after Maya manually creates them.

### Client response
1. If Maya asks a question or requests a document, switch back to **Manohar** or **Alex**.
2. The client home page now shows the newly created action.
3. Open **Questions**, **Documents**, or **Messages** depending on the request.
4. Answer the question, or use the simulated upload button on the requested document.
5. No real file is uploaded — the button briefly shows “Uploading…”, then updates the relevant fake document record and the workflow state.
6. The request becomes completed and responsibility returns to the Tax Preparer.

### Reviewer approval
1. Switch back to **Maya**.
2. Use **Submit for review** on the Review tab.
3. Switch to **Daniel Kim** (Reviewer).
4. Open the return from the **Review Queue** — it appears at the top with a **New** indicator.
5. Review the AI-supported findings, manual corrections, source evidence, preparer explanations, and open items.
6. Daniel can approve an item, request changes, request more information, or approve the complete return.
7. If Daniel requests changes/information, repeat the client-response workflow as needed.
8. When Daniel approves, the status becomes **“Waiting for client approval.”**

### Client approval and filing
1. Switch back to **Manohar** or **Alex**.
2. Open the return summary.
3. Review the main income, deductions or business totals, and the estimated refund or amount due.
4. Approve the return.
5. The status becomes **“Ready to file.”**
6. Switch to **Maya**, run the final check, and perform the simulated filing action.
7. The status becomes **“Filed.”**
8. The updated status is synchronized across the client view, tax preparer view, reviewer queue, firm administrator dashboard, and the return’s activity history.

## 5. How the Prototype Addresses the Ten Challenges

### 1. Source Document Traceability
Return fields connect to their extracted values, the source document, the exact page or section, and supporting evidence. Calculated fields show their transformation (the inputs that produced the total), and corrections/verifications are recorded on the field. Document parsing and OCR are simulated — the “extracted” values are seeded, not read from real files.

### 2. Client and CPA Collaboration
Messages and requests are attached to the specific document, question, issue, or return field they concern. Each request shows who owns the next action (client or firm). Client-visible messages are separated from internal firm notes, which clients can never see. Conversations and requests appear only when a user creates them, and they remain available for later reference.

### 3. Where to Start
New individual and business clients see onboarding first. The client home presents one clear primary action at a time, asks only a few questions, and reveals documents and features only when they are relevant. The intent is that a user can understand what to do within roughly ten seconds.

### 4. Getting Lost Between Parts of the App
Global navigation is separated from return-specific navigation. Return pages carry a header and context, and documents, questions, requests, messages, issues, and return fields are linked to one another. A user can open a piece of evidence and return to where they were. Dashboard and list filters are preserved as you move around.

### 5. Role-Aware Experiences
Navigation and available actions differ by role: the Individual Taxpayer and Business Owner see a client experience (home, return, documents, questions, messages); the Tax Preparer sees a dashboard, returns, tasks, and documents; the Reviewer sees a review queue and returns; the Firm Administrator sees firm overview, team, assignments, workload, deadlines, and permissions. Permissions prevent clients from seeing internal notes and prevent unauthorized roles from performing reviewer-only actions. Five roles are demonstrated in the prototype.

### 6. Return Status and Progress
Every return communicates its current stage, completed work, next step, responsible role, blocker (if any), deadline, and activity history. The same shared return state drives what each role sees, so a status change is reflected everywhere at once.

### 7. Actionable Dashboard
The Tax Preparer dashboard is a ranked work queue highlighting new returns, urgent and overdue work, blocked returns, waiting-on-client items, and returns ready for review, with the responsible person shown. The Administrator overview surfaces returns by stage, staff workload, urgent deadlines, and returns needing assignment. Generated fake returns and tasks are used to demonstrate sorting, filtering, prioritization, and scale.

### 8. Clickable vs. Editable
Field states are visually distinct: editable, read-only calculation, AI-generated, needs verification, verified, manually corrected (review required), approval required, and locked. Restricted actions are disabled with a reason where appropriate (for example, only preparers can classify documents; locked fields must be reopened before editing).

### 9. Complexity Made Navigable
The prototype uses progressive disclosure, summary-and-detail views, global search, list filtering, contextual tabs, and drawers/detail pages. A small number of returns are modeled deeply, while a larger set of generated returns, documents, tasks, and warnings demonstrates behaviour at scale. Results respect the viewer’s role and permissions.

### 10. Trustworthy AI
AI recommendations show what was detected, why it matters, the supporting evidence, a plain-language confidence/uncertainty statement, the recommended action, and alternative actions, along with accept / correct / dismiss / escalate controls. The AI never silently replaces a verified value — a human always makes the final decision, and dismissals require a reason.

## 6. What Is Genuinely Wired Up

The following are implemented and interactive in the prototype:

- Demo role switching (login screen + workspace switcher for Maya’s dual role)
- Individual Taxpayer onboarding
- Business Owner onboarding
- Conditional document/question relevance driven by onboarding answers
- Shared return status and progress across all roles
- Tax Preparer dashboard (ranked queue) and Returns list
- AI-assisted review interactions: accept, correct value, dismiss (with reason), ask client, request document, escalate
- Supporting-evidence and source-document traceability views (by field and by document)
- Field corrections with a manual-correction / reviewer-approval flow
- Client questions created by the preparer
- Document requests created by the preparer
- Contextual message threads linked to documents, issues, or fields
- Internal firm notes hidden from clients
- Reviewer queue with approve, request-changes, and item-level actions
- Client return approval
- Simulated final check and filing
- Firm Administrator visibility: firm overview, team, assignments, workload, deadlines, permissions
- Global search (⌘/Ctrl + K) and list/dashboard filtering
- Reset demo data
- Local persistence across reloads (browser localStorage)
- Audit/activity recording, surfaced in the return’s Overview “recent activity”

> Some return sub-tabs (Issues, Tasks, Activity as standalone pages) are placeholders; issues, tasks, and recent activity are surfaced on the Overview and, for tasks, on the preparer’s global Tasks page.

## 7. What Is Simulated

The following are intentionally simulated:

- Authentication and login (no password; personas are selected)
- User identity and permissions backend
- Database
- API responses
- Document upload and file storage
- OCR and document parsing
- AI recommendations and confidence values
- Tax calculations
- Tax-law validation
- Electronic signatures
- Tax-authority filing
- Email and external notifications
- Production security and compliance controls

Fake data and simulated processing were used deliberately so the case study could focus on frontend UX, interaction design, information architecture, and complete clickable workflows rather than backend infrastructure.

**Extensibility note:** the role and workflow architecture is generic and can be extended to additional roles (for example, seasonal staff) and to real services (authentication, storage, OCR, and an actual AI model) without reworking the frontend structure.

## 8. Important Design Decisions

- Onboarding appears before the return dashboard for new clients.
- Onboarding answers determine which documents and tax topics are relevant.
- Client screens reveal complexity gradually, one primary action at a time.
- No questions, requests, or conversations appear until someone creates them.
- The same return status is shared across every role rather than duplicated per screen.
- AI lives inside the Review workflow instead of a separate destination.
- Messages are connected to documents, fields, or issues rather than being a generic inbox.
- Internal notes are strictly hidden from clients.
- Fake uploads use a simple button to keep the demonstration fast.
- A few deeply modeled demo returns are combined with generated “scale” data for dashboards and lists.

## 9. Technology and Architecture


- **React 19** with **TypeScript**
- **Vite** (dev server and production build)
- **React Router** (routing, nested return-context routes)
- **Tailwind CSS** (styling)
- **Zustand** (application/domain state and stores)
- **Lucide** (icons)
- **Vitest** + **React Testing Library** (unit/component tests)
- **Playwright** (end-to-end tests)
- **oxlint** (linting)
- Browser **localStorage** (persistence)
- Seeded fake-data generators for scale data

Architecture in brief:

- **Feature-based organization** under `src/features/*` (clients, documents, messages, return-review, traceability, reviewer, dashboard, administration, etc.), with shared UI in `src/components/*`.
- **Shared domain state** in `src/store/*` (demo state, auth, onboarding, questionnaire, traceability, dashboard/list filters).
- **Simulated service layer** in `src/services/*` (status, collaboration, reviewer, preparer, document, AI, checklist, priority, admin, search) that encapsulates workflow rules.
- **Role and permission handling** via an auth store, route guards, and a permission gate component.
- **Seeded fake data** in `src/data/*` (named demo personas and returns plus a generator for larger datasets), persisted to localStorage with a version key so schema changes reset cleanly.

## 10. Running the Project

Requires Node.js 20.19+ (Node 22 recommended).

```bash
# install dependencies
npm install

# start the dev server (http://localhost:5173)
npm run dev

# production build and preview
npm run build
npm run preview

# lint
npm run lint

# unit / component tests
npm run test

# end-to-end tests (Playwright)
npm run test:e2e
```

On first load, sign in as any of the five demo personas. Use the **Reset demo data** button in the top bar to return everything to its initial state at any time.
